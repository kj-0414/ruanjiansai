# 问题日志

## 1. 多角色支持功能

### 问题描述
用户希望一个账号可以同时拥有求职者和企业两种身份，自由切换，无需重新登录。

### 问题原因
原系统设计为单一角色模式：
- 数据库 `User` 表的 `role` 字段只能存储一个角色
- 登录时需要选择角色，且登录后无法切换

### 解决方案

#### 数据库修改
**文件**: `backend/core/database.py`

```python
# 修改前
role = Column(String, nullable=False)  # job_seeker or company

# 修改后
roles = Column(Text, nullable=False)  # JSON string, e.g., ["job_seeker", "company"]
```

#### 后端修改
**文件**: `backend/api/user.py`

- 注册时默认赋予两种角色: `json.dumps(["job_seeker", "company"])`
- 登录和获取用户信息时返回角色列表
- 修复 `roles` 为空时访问 `roles[0]` 导致的错误

#### 前端修改
**文件**: `frontend/src/services/auth.js`

- `login()`: 存储 `roles` 到 localStorage
- `getAllRoles()`: 默认返回 `['job_seeker', 'company']`
- `switchRole()`: 切换角色时同时更新 user 对象

**文件**: `frontend/src/layouts/MainLayout.vue`

- 添加角色切换下拉菜单
- 页面加载时从 localStorage 读取角色，不从后端获取

**文件**: `frontend/src/router/guard.js`

- 移除路由切换时调用 `profile()` 的逻辑
- 只验证 token 是否存在，避免后端返回的角色覆盖本地存储的角色

### 核心流程
1. 用户注册 → 自动获得两种角色
2. 用户登录 → 输入手机号和密码，无需选择角色
3. 角色切换 → 在导航栏下拉菜单中选择身份
4. 页面刷新 → 使用 localStorage 中的角色，不会被后端数据覆盖
5. 权限验证 → 后端检查用户是否拥有请求的角色

## 2. 角色切换后闪回问题

### 问题描述
用户切换到企业角色后，页面刷新后又闪回到求职者角色。

### 问题原因
**路由守卫**（`frontend/src/router/guard.js`）在每次路由切换时都会调用 `authService.profile()`，从后端获取用户信息。后端返回的永远是数据库中的原始角色（第一个角色），这会覆盖 localStorage 中存储的切换后的角色。

### 解决方案

#### 修改路由守卫
**文件**: `frontend/src/router/guard.js`

```javascript
// 修改前
try {
  await authService.profile()  // 这会覆盖 localStorage 中的角色
  next()
} catch (error) {
  ElMessage.error('登录已过期，请重新登录')
  next('/login')
}

// 修改后
// 只验证 token 是否存在，不获取完整的用户信息
// 避免后端返回的角色覆盖本地存储的角色（支持角色切换）
next()
```

#### 修改 MainLayout 初始化逻辑
**文件**: `frontend/src/layouts/MainLayout.vue`

页面加载时直接使用 localStorage 中的角色数据，不从后端获取：
```javascript
const userData = authService.getCurrentUser()
if (userData) {
  const currentRole = authService.getCurrentRole()
  user.value = {
    ...userData,
    role: currentRole  // 使用本地存储的角色
  }
}
```

## 关键教训

1. **前端状态 vs 后端状态**：角色切换是前端行为，不应该被后端数据覆盖
2. **路由守卫的副作用**：路由守卫中调用 API 可能产生意外的副作用
3. **localStorage 的作用**：客户端存储可以用于保存用户的临时状态（如当前角色）
4. **组件初始化顺序**：页面加载时组件的初始化顺序会影响最终显示的角色
